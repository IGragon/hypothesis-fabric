# Product Proposal — Вариант 2 (Hybrid RAG + agent)

> Архитектурная позиция: **детерминированный retrieval-first каркас с LLM в
> хорошо определённых слотах**. Оркестратор — стейт-машина; LLM работает в
> рамках слотов (реранк, генерация, качественная оценка). Опирается на
> `analysis.md`.

---

## 1. Обоснование идеи

Прикладная задача: исследователь получает инструмент, где **воспроизводимый
каркас поиска и оценки** объединён с **креативностью LLM в генерации гипотез**.
Система предbuilt-индексирует корпус (текст/PDF/Excel/БД) и строит граф знаний
детерминированно [R-F1..4]; на запрос цели/ограничений [R-IN1..2] проходит через
фиксированные стадии: retrieve → generate → score → explain → export. LLM
включается только в слотах, где правила слабы (переранжирование evidence,
генерация гипотез по аналогиям/контрфактуалам, качественная оценка осей) [R-F5][R-F6][R-F7].

Позиция V2 оправдана там, где нужна **связка гибкости и воспроизводимости**:
детерминированный каркас даёт интерпретируемость [R-N1], контроль контекста
[R-N4] и масштабируемость на новые домены [R-K4]; LLM-слоты — новизну гипотез,
недостижимую чистыми правилами. Canvas-модель (re-run стадии) [R-F14] прямо
соответствует видению из `.ignore/DESIGN.md`.

Где LLM **не привлекается**: парсинг/индекс, query plan, привязка цитат, проверка
ограничений, взвешенное ранжирование, экспорт — всё детерминировано [R-F9][R-IN2][R-F12].

## 2. Цель и метрики

### 2.1 Цель проекта

За один интерактивный запуск (минуты) генерировать ранжированный набор
проверяемых гипотез с обоснованием, цитатами и оценкой, с воспроизводимыми
стадиями и контролем контекста [R-N4][R-K1][R-N1].

### 2.2 Метрики успеха

| Категория | Метрика | Цель | Требование |
|-----------|---------|------|------------|
| **Продуктовая** | Доля гипотез, признанных экспертом «проверяемыми» | ≥ 65% | [R-K1] |
| Продуктовая | Coverage цитат | ≥ 85% | [R-K2][R-F9] |
| Продуктовая | Доля гипотез, прошедших constraint_check | 100% (hard gate) | [R-IN2] |
| Продуктовая | Воспроизводимость ранжирования (Jaccard@10 при re-run без фидбэка) | ≥ 0.9 | [R-N1] |
| **Агентская** | Доля run, дошедших до Export без FE5 | ≥ 95% | [R-N4] |
| Агентская | Hallucination-source rate | ≤ 4% | [R-K2] |
| **Техническая** | Время до первичного набора (p50 / p95) | ≤ 2 мин / ≤ 5 мин | [R-N4] |
| Техническая | Токены/сессия (input/output) | ≤ 20k / 6k | [R-N4] |
| Техническая | Доступность API | ≥ 99.5% | [R-N4] |
| Техническая | Локализация данных (on-prem режим) | 100% | [R-N5] |
| Техническая | ETL latency (индекс корпуса, batch) | ≤ 2 ч на 10⁴ док. [ASSUM-1] | [R-F1] |

## 6. Потенциальный data flow и делегирование LLM/Agent vs deterministic

### 6.1 Data flow (кратко)

База знаний → batch ETL → VectorDB + KG (предbuilt). Пользователь → цель/
ограничения → Orchestrator → Retrieve (query plan → vector+KG → LLM rerank) →
Generate slot (кандидаты) → Citation Bind → Score (features → LLM Judge →
Weighted Ranker) → Explain slot → Export. Память: Session Store (run) +
Feedback Labels (долгосрочная). Полная схема — `05-dataflow.md`.

### 6.2 Что делегируется `LLM/Agent` vs `deterministic`

| Этап | `LLM/Agent` | `deterministic` | Обоснование |
|------|-------------|-----------------|-------------|
| Парсинг/индекс/KG | — | ✅ | воспроизводимость [R-F1..4] |
| Query plan retrieval | — | ✅ | детерминированность [R-IN1][R-IN2] |
| Реранжирование evidence | ✅ | — | семантическая релевантность [R-F7] |
| Генерация гипотез | ✅ | — | аналогии/контрфактуалы [R-F6] |
| Выявление пробелов | ✅ | — | плохо формализуется [R-F5] |
| Привязка цитат | — | ✅ | точность [R-K2][R-F9] |
| Признаки скоринга | — | ✅ | граф.дистанция/constraint-match [R-OUT5..7] |
| Качественная оценка осей | ✅ | — | семантика рисков/эффекта [R-F7] |
| Взвешенное ранжирование | — | ✅ | формула, веса [R-A1] |
| Проверка ограничений | — | ✅ | безопасность [R-IN2] |
| Обоснование (текст) | ✅ | — | NL-генерация [R-F8] |
| Визуализация связей | — | ✅ | рендер графа [R-F10] |
| Экспорт | — | ✅ | форматирование/REST [R-F12][R-F13] |
| Калибровка по фидбэку | — | ✅ | пере-взвешивание [R-A3][ASSUM-7] |

### 6.3 Разметка безопасности автономии

- **LLM в слотах, но не у руля**: LLM не решает, какую стадию запускать; переходы
  — детерминированные правила (#3 Transition Rules). Это снижает риск зацикливания
  и неконтролируемого расхода токенов [R-N4][R-N1].
- **Hard gates детерминированы**: citation bind, constraint_check, weighted
  ranker — не делегируются LLM [R-K2][R-IN2].
